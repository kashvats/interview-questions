Real-world operational rules
Migrations
Run them as a job, not hidden inside every pod start.
Celery beat
Keep one replica unless you deliberately design leader election.
Workers
Scale workers separately from web pods.
Backend autoscaling
Use HPA on API pods.
Frontend autoscaling
Usually lower priority, since it serves static files.
Secrets
Use secret manager integration, not plain files in git .

Important production note about migrations
In strict production setups, migrations are often run as a separate job, not every app startup.
This entrypoint is still acceptable for many deployments, but the more mature pattern is:
app container starts Gunicorn

STEP 6 ENVIRONMENT MANAGEMENT WITH KUSTOMIZE
These are the decisions you will be asked to justify in staff+ interviews,
architecture reviews, and production incident post-mortems.
💡  ARCHITECTURE REASONING
Why replicas: 3 minimum for backend:
SOURCE CODE / MANIFEST
  - 1 pod is a single point of failure.
  - 2 pods: one node drain takes you to 1 pod (PDB allows it if minAvailable: 1).  - 3 pods with maxUnavailable: 0 and PDB minAvailable: 2 ensures no node drain
    or rolling update ever drops below 2 running pods simultaneously.
💡  ARCHITECTURE REASONING
Why maxUnavailable: 0 and maxSurge: 1:
SOURCE CODE / MANIFEST
  - maxUnavailable: 0 means no pod is terminated before a replacement is healthy.
  - maxSurge: 1 creates one extra pod above desired count during rollout.
  - Result is zero-downtime rolling update, at the cost of needing extra capacity    for 1 pod during the window. Rollout is sequential, one pod at a time.
💡  ARCHITECTURE REASONING
Why managed Postgres over self-hosted in Kubernetes:
SOURCE CODE / MANIFEST
  - Stateful systems in Kubernetes require persistent volumes, replication
    configuration, backup automation, and careful pod disruption management.
  - AWS RDS provides automatic failover, backups, point-in-time recovery,    encryption, security patching, and Performance Insights out of the box.
  - Self-managing Postgres in Kubernetes is rarely worth the operational cost
    unless you are at very large scale or have unusual compliance requirements.
💡  ARCHITECTURE REASONING
Why readOnlyRootFilesystem: true:
SOURCE CODE / MANIFEST
  - A read-only root filesystem means an attacker who achieves code execution    cannot write malicious files to the container filesystem.
  - The app can still write to explicitly mounted emptyDir volumes like /tmp.
  - This is a CIS Kubernetes Benchmark requirement. Most security audits
    and SOC 2 reviews will flag its absence.
💡  ARCHITECTURE REASONING
Why terminationGracePeriodSeconds: 30:
SOURCE CODE / MANIFEST
  - When Kubernetes sends SIGTERM, the app should drain in-flight requests.
  - The preStop sleep hook delays shutdown by 5 seconds to give the load
    balancer time to stop routing new traffic before SIGTERM is sent.
  - If your app drains in more than 30s, increase this value. Kubernetes    sends SIGKILL unconditionally after terminationGracePeriodSeconds.
💡  ARCHITECTURE REASONING
Why replicas: 1 for Celery Beat:
SOURCE CODE / MANIFEST
  - Celery Beat is a scheduler. Two instances simultaneously enqueue the same
    periodic task twice, causing double-execution that can corrupt data.
  - Use django-celery-beat with DatabaseScheduler so the schedule is stored in    Postgres and can be modified without redeploying beat.
💡  ARCHITECTURE REASONING
Why separate Deployments for worker and beat:
SOURCE CODE / MANIFEST
  - Different scaling requirements: HPA-scale workers but keep beat at exactly 1.
  - Separate Deployments give separate resource limits, separate rollout history,    and separate monitoring targets. Coupling them adds operational complexity.
💡  ARCHITECTURE REASONING
Why topologyKey: kubernetes.io/hostname for pod anti-affinity:
SOURCE CODE / MANIFEST
  - Without anti-affinity, the scheduler may place all backend pods on the same
    node. A single node failure then kills all replicas simultaneously.
  - Preferred anti-affinity (not required) tells the scheduler to spread pods    across different nodes without blocking scheduling if only one node exists.
💡  ARCHITECTURE REASONING
Why External Secrets Operator over SealedSecrets or plain Secrets:
SOURCE CODE / MANIFEST
  - Plain Secrets committed to git expose credentials to every git user.
  - SealedSecrets requires re-encrypting and committing every time a secret rotates.
  - ESO with SSM: rotate the value in SSM, ESO picks it up within refreshInterval.    No git commit, no re-encryption, no manual intervention needed.
💡  ARCHITECTURE REASONING
Why IRSA instead of IAM user credentials for S3 and SSM access:
SOURCE CODE / MANIFEST
  - IAM user access keys are long-lived static credentials. If leaked via logs,
    environment dumps, or git history, they remain valid until manually rotated.
  - IRSA issues short-lived tokens to pods via Kubernetes service account OIDC.    Tokens expire every 15-60 minutes and are scoped to the specific SA.
    No keys to store, no rotation to manage, no leakage risk in CI log output.
💡  ARCHITECTURE REASONING
Why transit_encryption_enabled for Redis:
SOURCE CODE / MANIFEST
  - Celery tasks can contain user data, PII, or business-critical payloads.
  - Unencrypted Redis in the same VPC is an internal threat vector if any other
    service or node is compromised. TLS in transit is cheap and mandatory.
💡  ARCHITECTURE REASONING
Why PDB minAvailable: 2 for backend:
SOURCE CODE / MANIFEST
  - Cluster upgrades, spot interruptions, and voluntary node drains all trigger
    voluntary disruptions. Without a PDB, Kubernetes can evict all backend pods
    from a node at once during a drain.
  - minAvailable: 2 ensures at least 2 pods remain healthy during any voluntary    disruption, keeping the service operational through maintenance windows.

--- 1.1 Migration Job ---
A migration Job is a one-shot task that runs Django migrate before pods roll.
It uses the same backend image but a different command.
Save as: infra/kubernetes/job-migrate.yaml
SOURCE CODE / MANIFEST
apiVersion: batch/v1                       # Batch API group for one-off tasks.
kind: Job                                  # A Job runs to completion, not forever.
metadata:  name: backend-migrate                    # Job name.
  namespace: production-app               # Same namespace as app pods.
  labels:
    app: backend                           # Label for querying and selection.    component: migration                   # Mark this as a migration job.
spec:
  backoffLimit: 2                          # Retry up to 2 times before marking Failed.
  ttlSecondsAfterFinished: 600            # Auto-delete 10 min after completion.  template:
    spec:
      restartPolicy: OnFailure             # Retry the pod on failure, not on success.
      serviceAccountName: app-sa           # Same SA as app containers.
      securityContext:        runAsNonRoot: true                 # Non-root required.
        seccompProfile:
          type: RuntimeDefault             # Default seccomp restrictions.
      containers:        - name: migrate
          image: ghcr.io/your-org/backend:latest   # Same backend image.
          command: ["python", "manage.py", "migrate", "--noinput"]  # Only run migrations.
          envFrom:            - secretRef:
                name: app-secrets          # DB credentials come from the Secret.
            - configMapRef:
                name: app-config           # Non-secret config from ConfigMap.          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
            limits:              cpu: "500m"
              memory: "256Mi"
          securityContext:
            allowPrivilegeEscalation: false            readOnlyRootFilesystem: true
            capabilities:
              drop:
                - ALL
💡  ARCHITECTURE REASONING
Why this matters (staff-level reasoning):
Running migrations as a Job instead of in every pod startup prevents race
conditions when you have 3+ replicas all trying to migrate simultaneously.
The ttlSecondsAfterFinished cleans up the completed pod automatically so
you do not accumulate stale Job objects in the cluster.

💡  ARCHITECTURE REASONING
Why use ESO instead of SealedSecrets or plain Secrets:
ESO pulls the live value from SSM every refreshInterval. When you rotate in
SSM, ESO automatically updates the Kubernetes Secret. SealedSecrets requires
re-encrypting and re-committing. Plain Secrets in git  are never acceptable.

STEP 9 ADVANCED INGRESS & NETWORKING
A staff engineer owns the runbook. Here is the baseline operational playbook.
--- 9.1 Pod crash loop ---
Symptoms:
SOURCE CODE / MANIFEST
  kubectl get pods -n production-app  STATUS: CrashLoopBackOff
Diagnosis:
SOURCE CODE / MANIFEST
  kubectl describe pod <pod-name> -n production-app
  kubectl logs <pod-name> -n production-app --previous
Common causes:
1. Missing environment variable (DB URL, Redis URL not set in Secret)
2. Failed migration (table does not exist, schema mismatch)
3. OOMKilled (memory limit too low for the workload)
4. Liveness probe timing out before app starts (wrong probe path or threshold)
Fix for OOMKilled:
SOURCE CODE / MANIFEST
  kubectl get pod <pod-name> -n production-app \
    -o jsonpath="{.status.containerStatuses[*].lastState}"
  Look for reason: OOMKilled, then increase memory limit in deployment spec.
--- 9.2 Deployment rollback ---
If a bad deploy goes out immediately roll back:
SOURCE CODE / MANIFEST
  kubectl rollout history deployment/backend -n production-app
  kubectl rollout undo deployment/backend -n production-app  kubectl rollout undo deployment/backend -n production-app --to-revision=3
  kubectl rollout status deployment/backend -n production-app
Using Helm:
SOURCE CODE / MANIFEST
  helm rollback backend 2 -n production-app
--- 9.3 Database connection pool exhaustion ---
Symptoms:
SOURCE CODE / MANIFEST
  Django logs: remaining connection slots are reserved for non-replication superuser
  Or: FATAL: sorry, too many clients already
Diagnosis:
SOURCE CODE / MANIFEST
  Check open connections on RDS:    SELECT count(*) FROM pg_stat_activity WHERE datname = 'your_db';
Root cause:
Django + Gunicorn without a connection pooler can exhaust Postgres limits.
Each Gunicorn worker holds one connection per active request.
SOURCE CODE / MANIFEST
  3 pods x 9 workers x 2 threads = 54 potential simultaneous connections.
Fix:
SOURCE CODE / MANIFEST
  Add PgBouncer in transaction pooling mode as a sidecar:
  containers:
    - name: backend
      image: ghcr.io/your-org/backend:latest      env:
        - name: DATABASE_URL
          value: postgresql://user:pass@localhost:5432/db
    - name: pgbouncer      image: edoburu/pgbouncer:latest
      env:
        - name: DATABASE_URL
          value: postgresql://user:pass@your-rds-host:5432/db        - name: POOL_MODE
          value: transaction
        - name: MAX_CLIENT_CONN
          value: "200"
--- 9.4 HPA not scaling ---
Symptoms:
High CPU observed, but replica count stuck at minimum.
Diagnosis:
SOURCE CODE / MANIFEST
  kubectl describe hpa backend-hpa -n production-app
  Look for: unable to fetch metrics or metric not available
Common causes:
1. metrics-server not installed
2. Resource requests not set on the container (HPA needs requests to derive utilization ratio)
3. CPU actually not as high as expected once normalized against requests
Fix:
SOURCE CODE / MANIFEST
  Install metrics-server:    kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
  Verify resource requests are set:
    kubectl get deployment backend -n production-app \
      -o jsonpath="{.spec.template.spec.containers[0].resources}"
--- 9.5 Celery task queue backing up ---
Symptoms:
Slow task processing. Queue length growing.
SOURCE CODE / MANIFEST
  Check queue length: redis-cli -h <redis-host> llen celery
Diagnosis:
SOURCE CODE / MANIFEST
  Check worker logs:
    kubectl logs -l app=celery-worker -n production-app --tail=100
Immediate fix:
SOURCE CODE / MANIFEST
  kubectl scale deployment celery-worker --replicas=6 -n production-app
Root cause investigation:
Is there a task hanging? Check for stuck ETA tasks.
Is a downstream service (RDS, Redis, external API) slow or throttled?
Are tasks retrying on failure and filling the queue?

[ ] All tests pass in CI.

[ ] Migration tested on a staging environment first.

    options {                                                               // Global pipeline options.

            }
        }
        stage('Django checks') {                                            // Run framework-level validation.            steps {
                sh '''
                    . .venv/bin/activate                                    # Activate the virtual environment.

            }
        }
        stage('Tests') {                                                    // Run automated test suite.
            steps {
                sh '''
                    . .venv/bin/activate                                    # Activate the virtual environment.                    python manage.py test                                   # Run Django tests.
                '''
            }
        }
        stage('Build Docker image') {                                       // Build the production Docker image.
            steps {
                sh '''
                    docker build -t ${IMAGE_FULL} -t ${IMAGE_LATEST} .      # Build image with versioned and latest t                '''
            }
        }
        stage('Push Docker image') {                                        // Authenticate and push to registry.

STEP 16 COST OPTIMIZATION (SPOT & KARPENTER)
Running "pythonmanage.py migrate" on deploy sounds simple. In production it
causes table locks, breaks running pods, and can corrupt data if the migration
is not designed carefully. This step explains the safe pattern: expand/contract.
--- 12.1 The problem with naive migration ---
Assume you have version V1 of your app running with 3 pods. You deploy V2 which
renames a column from "user_name" to "username".
What happens during rolling update:
SOURCE CODE / MANIFEST
  - V2 pod starts. Django migrate renames the column.
  - V1 pods are still running and querying "user_name" which no longer exists.
  - V1 pods start throwing 500 errors for every DB query.  - Rolling update takes 60-120 seconds. That is 60-120 seconds of production errors.
This is why naive migration is not zero-downtime.
--- 12.2 The expand/contract  pattern ---
The expand/contract  pattern splits any breaking migration into three safe phases:
Phase 1 - Expand (deploy with old + new column both working):
SOURCE CODE / MANIFEST
  - Add the new column as nullable. Do not drop the old one.  - Write application code that writes to BOTH columns.
  - Read from the old column.
  - Deploy. Zero downtime. Old and new app versions work.
Phase 2 - Migrate data:
SOURCE CODE / MANIFEST
  - Run a data migration to copy data from old column to new column.  - Do this as a background job or a separate management command, not inline.
  - Verify data parity in both columns.
Phase 3 - Contract (remove the old column):
SOURCE CODE / MANIFEST
  - Update app code to read from new column only.
  - Remove writes to old column.  - Deploy and verify.
  - Then run a migration to drop the old column.
This means every breaking database change takes 3 deployments.
That is the correct cost of zero-downtime production operations.
--- 12.3 Django expand/contract  example ---
Phase 1 migration - expand:
SOURCE CODE / MANIFEST
  class Migration(migrations.Migration):      dependencies = [('users', '0012_previous')]
      operations = [
          migrations.AddField(
              model_name='user',              field=models.CharField(
                  db_column='username',
                  max_length=150,
                  null=True,     # Nullable. No lock on existing rows.                  blank=True,
              ),
          ),
      ]
Phase 1 model code - write to both:
SOURCE CODE / MANIFEST
  class User(models.Model):
      user_name = models.CharField(max_length=150)    # Old column. Keep for now.
      username = models.CharField(max_length=150, null=True, blank=True)
      def save(self, *args, **kwargs):          self.username = self.user_name   # Dual-write during transition.
          super().save(*args, **kwargs)
Phase 2 - data migration management command:
SOURCE CODE / MANIFEST
  from django.core.management.base import BaseCommand
  from users.models import User
  class Command(BaseCommand):
      help = 'Backfill username from user_name'
      def handle(self, *args, **kwargs):
          batch_size = 1000
          qs = User.objects.filter(username__isnull=True)
          total = qs.count()          updated = 0
          while True:
              batch = list(qs[:batch_size])
              if not batch:                  break
              for user in batch:
                  user.username = user.user_name
              User.objects.bulk_update(batch, ['username'])
              updated += len(batch)              self.stdout.write(f'Updated {updated}/{total}')
Run as a Kubernetes Job:
SOURCE CODE / MANIFEST
  kubectl create job backfill-username \
    --image=ghcr.io/your-org/backend:latest \
    -- python manage.py backfill_username
Phase 3 migration - contract (after all pods on V3):
SOURCE CODE / MANIFEST
  class Migration(migrations.Migration):
      operations = [
          migrations.RemoveField(model_name='user', name='user_name'),
      ]
--- 12.4 Table locking reference by migration type ---
Migration type Lock type Safe in prod?
AddField (nullable) No table lock Yes - safe to run online
AddField (non-nullable) Full table lock No - use expand/contract
RemoveField No table lock Yes - after code is removed
RenameField Full table lock No - use expand/contract
AlterField (add index) Full table lock No - use CREATE INDEX CONCURRENTLY
AlterField (change type) Full table lock No - use expand/contract
AddIndex (regular) Full table lock No - use RunSQL with CONCURRENTLY
--- 12.5 Creating indexes without locking ---
A regular CREATEINDEX locks the table for reads and writes.
CREATEINDEXCONCURRENTLY  does not, but Django's default AddIndex uses the
locking version. Override it with RunSQL:
SOURCE CODE / MANIFEST
  class Migration(migrations.Migration):
      atomic = False   # Required for CONCURRENTLY.
      operations = [
          migrations.RunSQL(              sql="CREATE INDEX CONCURRENTLY IF NOT EXISTS "
                  "users_user_email_idx ON users_user (email)",
              reverse_sql="DROP INDEX IF EXISTS users_user_email_idx",
          ),
      ]
💡  ARCHITECTURE REASONING
Why atomic = False:
CONCURRENTLY  cannot run inside a transaction. Django wraps migrations in
SOURCE CODE / MANIFEST
  transactions by default. Setting atomic = False disables that for this
  migration only.
--- 12.6 Migration job in the deploy pipeline ---
Correct order for every production deploy:
SOURCE CODE / MANIFEST
  Step 1: Run migration Job.
  Step 2: Wait for Job to complete successfully.  Step 3: Roll out new Deployment.
Never rollout before migration. Never run migration inline in pod startup
when you have multiple replicas. Migration Job runs once. Rolling update
starts after.
In GitHub Actions:
SOURCE CODE / MANIFEST
  - name: Run migration    run: |
      kubectl delete job backend-migrate -n production-app --ignore-not-found
      kubectl apply -f infra/kubernetes/job-migrate.yaml
      kubectl wait --for=condition=complete job/backend-migrate \        -n production-app --timeout=600s
  - name: Roll out backend
    run: |      kubectl set image deployment/backend backend=ghcr.io/your-org/backend:$TAG \
        -n production-app
      kubectl rollout status deployment/backend -n production-app --timeout=300s
--- 12.7 What to do when a migration fails in prod ---
SOURCE CODE / MANIFEST
  Step 1: Immediately stop the rollout.    kubectl rollout undo deployment/backend -n production-app
  Step 2: Check migration status.
    kubectl logs job/backend-migrate -n production-app
  Step 3: Determine if migration partially applied.
    Check django_migrations table:
      SELECT * FROM django_migrations ORDER BY applied DESC LIMIT 10;
  Step 4: If safe to revert, run the reverse migration manually:
    kubectl run revert --image=ghcr.io/your-org/backend:PREVIOUS_TAG \
      --restart=Never -- python manage.py migrate users 0012_previous
  Step 5: Verify app is functional on old code.
  Step 6: Fix migration and redeploy.
Key rule: Never write irreversible migrations (like dropping a column with data)
without a tested restore plan and database snapshot taken before deploy.

STEP 19 STAFF SYSTEM DESIGN REVIEW
Everything in this guide works fine at 10 pods and 1000 requests/minute.
This step documents what breaks at 100 pods and 100,000 requests/minute —
which is exactly what gets asked at staff-level system design interviews.
--- 15.1 etcd pressure ---
etcd is the key-value store behind Kubernetes. Every object (Pod, Service,
ConfigMap, Secret, Event) is stored in etcd. At scale:
SOURCE CODE / MANIFEST
  Problem: Kubernetes Events are stored in etcd and expire after 1 hour.
  At high deploy frequency, Events accumulate faster than they are pruned.  etcd memory grows. At 2 GB etcd can become unstable.
  Problem: HPA checks metrics every 15 seconds and writes back decisions.
  With 50 HPAs all polling and writing, etcd write throughput becomes a
  bottleneck. Watch the apiserver_etcd_db_total_size_in_bytes metric.
  Fix: Limit Event retention.
    --event-ttl=30m on kube-apiserver (default is 1 hour).
  Fix: Use a separate etcd cluster for Events in large deployments.
  Fix: Reduce unnecessary kubectl apply calls in CI. Use Server-Side Apply
  (kubectl apply --server-side) which reduces write amplification.
--- 15.2 Image pull storms ---
Problem: You deploy a new image tag to 100 pods simultaneously. All 100 pods
try to pull the same 1 GB image from GHCR at the same second.
Each node pulls the image independently. If you have 20 nodes, 20 concurrent
pulls. Your image registry rate limits you or your network saturates.
Result: Pods stay in ContainerCreating for 5-10 minutes. Traffic drops.
Fixes:
SOURCE CODE / MANIFEST
  Fix 1: Use imagePullPolicy: IfNotPresent.    If the node already has the image (from a previous deploy), it will not pull.
    This means rolling updates only pull on nodes getting the image for the first
    time, not on every restart.
  Fix 2: Pre-pull images with a DaemonSet.
    Deploy a DaemonSet that pulls your image on every node before the rollout.
    kubectl set image ds/image-prepuller image=ghcr.io/your-org/backend:NEW_TAG
    Wait until all nodes have the image, then roll the Deployment.
  Fix 3: Use a local registry mirror on each node.
    Configure containerd to use a pull-through registry cache running on your
    NAT instance or a dedicated EC2. Nodes pull from local cache first.
  Fix 4: Reduce image size.
    Multi-stage builds already help. Also: use --mount=type=cache in Dockerfile,
    separate dependencies layer from code layer so only code layer changes on
    most deploys.
  Fix 5: Limit maxSurge to 1 or 2 during rollout.
    maxSurge: 1 means only 1 new pod is created at a time. Pulls happen serially
    instead of in parallel. Rollout is slower but stable.
--- 15.3 DNS throttling at scale ---
Problem: CoreDNS in Kubernetes handles all service discovery and external DNS.
At 100 pods, every pod makes constant DNS lookups: your-service.namespace.svc.cluster.local.
AWS limits DNS queries to 1024 per second per VPC. At 100 pods each making
10 DNS requests/second, you exceed this limit. Pods start seeing DNS timeouts.
This looks like intermittent connection errors to Postgres, Redis, and external APIs.
Fixes:
SOURCE CODE / MANIFEST
  Fix 1: Enable ndots:5 workaround.    By default Kubernetes resolves A.B.C.D by trying multiple search domains
    (A.B.C.D.namespace.svc.cluster.local, then A.B.C.D.svc.cluster.local, etc).
    This multiplies DNS queries by 5-6x for each lookup.
    Set ndots: 2 in Pod dnsConfig to reduce search attempts:
    spec:
      dnsConfig:
        options:
          - name: ndots            value: "2"
          - name: single-request-reopen
            value: ""
  Fix 2: Scale CoreDNS.
    CoreDNS defaults to 2 replicas. At scale, increase to 4-6.
    kubectl scale deployment coredns --replicas=6 -n kube-system
  Fix 3: Use fully qualified domain names in your app config.
    Instead of: redis
    Use: redis.production-app.svc.cluster.local
    This skips the search domain expansion entirely.
  Fix 4: Enable NodeLocal DNSCache.
    Run a DNS cache DaemonSet on every node. Pods query the node-local cache
    (169.254.20.10) instead of CoreDNS pods. Cache hit rate is 80-90%.
    This is the single highest-impact DNS fix at scale.    https://kubernetes.io/docs/tasks/administer-cluster/nodelocaldns/
--- 15.4 Control plane rate limiting ---
Problem: kubectl , CI pipelines, ArgoCD, HPA, and every controller all talk
to the Kubernetes API server. At scale they can exhaust the API server's
request quota (--max-requests-inflight default 400).
Symptoms: kubectl  get pods takes 10+ seconds. ArgoCD sync is slow.
HPA decisions are delayed.
Fixes:
SOURCE CODE / MANIFEST
  Fix 1: Use watch instead of polling.
    kubectl get pods --watch is much cheaper than running kubectl get pods
    in a loop. CI pipelines that poll every 5 seconds are anti-patterns.
  Fix 2: Limit ArgoCD sync frequency.
    Set app.kubernetes.io/managed-by and timeout.reconciliation: 3m
    to reduce unnecessary ArgoCD API calls.
  Fix 3: Use Server-Side Apply.
    kubectl apply --server-side reduces the diff computation load on the
    client and applies patches more efficiently on the server.
--- 15.5 Noisy neighbor pods ---
Problem: A single pod with a memory leak or CPU spike can starve other pods
on the same node. This is the "noisy neighbor" problem.
Fixes:
SOURCE CODE / MANIFEST
  Fix 1: Set accurate resource requests AND limits on every container.
    Without limits, one misbehaving pod can consume 100% of node CPU and
    cause CPU throttling for all other pods on the node.
  Fix 2: Use LimitRange to enforce defaults in the namespace.
    apiVersion: v1
    kind: LimitRange    metadata:
      name: default-limits
      namespace: production-app
    spec:
      limits:        - type: Container
          default:
            cpu: 500m
            memory: 256Mi          defaultRequest:
            cpu: 100m
            memory: 128Mi
          max:            cpu: "4"
            memory: 4Gi
  Fix 3: Use ResourceQuota to cap total namespace resource usage.
    apiVersion: v1
    kind: ResourceQuota
    metadata:      name: namespace-quota
      namespace: production-app
    spec:
      hard:        requests.cpu: "16"
        requests.memory: 16Gi
        limits.cpu: "32"
        limits.memory: 32Gi
        pods: "50"
  Fix 4: Use Pod QoS classes intentionally.
    Guaranteed QoS: requests == limits. Never evicted first. Use for backend.
    Burstable QoS: requests < limits. Medium eviction priority. Use for workers.    BestEffort QoS: no requests or limits. Evicted first. Never use in prod.
--- 15.6 Long GC pauses in JVM (Java/Kotlin/Scala services) ---
This does not apply to Django, but staff engineers often oversee multi-language
stacks. JVM services under load hit stop-the-world GC pauses.
Liveness probe failures during GC pauses cause Kubernetes to restart the pod
(thinking it is unhealthy). This makes GC pauses compound into pod restarts.
Fix: Use startupProbe with a long failureThreshold to allow the JVM to warm up.
Fix: Use G1GC or ZGC with tuned max pause targets (-XX:MaxGCPauseMillis=50).
Fix: Increase initialDelaySeconds on livenessProbe for JVM services to 60+.

--- 16.2 What the API Gateway handles (not the app) ---
Authentication: Verify JWT or API key before request reaches Django.
Django does not need to re-verify on every endpoint.
Rate Limiting: Limit each API key or IP to 1000 req/min.
Without this, one customer can DDoS your entire backend.
Request Routing: /api/v1/  goes to old pods. /api/v2/  to new pods.
Zero-code A/B routing or versioned API rollouts.
Request Logging: Log every request with trace ID at the gateway level.
Uniform audit log across all services.
Request Transformation: Add Correlation-ID header to every request.
Strip internal headers before they reach clients.
Throttling and quotas: Per-plan limits for SaaS billing models.
SSL Termination: Terminate TLS at the gateway. Backend pods get plain HTTP .
--- 16.3 Kong on Kubernetes ---
Kong is the most common open-source API Gateway for Kubernetes.
It runs as a pod and integrates via CRDs.
Install:
SOURCE CODE / MANIFEST
  helm repo add kong https://charts.konghq.com
  helm upgrade --install kong kong/ingress \
    --namespace kong --create-namespace \
    --set controller.ingressClass=kong
Rate limiting plugin:
SOURCE CODE / MANIFEST
  apiVersion: configuration.konghq.com/v1
  kind: KongPlugin
  metadata:
    name: rate-limit-backend    namespace: production-app
  config:
    minute: 1000               # 1000 requests per minute per consumer.
    policy: redis              # Use Redis for distributed counting.
    redis_host: redis.production-app.svc.cluster.local    redis_port: 6379
  plugin: rate-limiting
Apply plugin to the Ingress:
SOURCE CODE / MANIFEST
  apiVersion: networking.k8s.io/v1
  kind: Ingress
  metadata:
    name: backend-ingress    annotations:
      konghq.com/plugins: rate-limit-backend
      kubernetes.io/ingress.class: kong
JWT authentication plugin:
SOURCE CODE / MANIFEST
  apiVersion: configuration.konghq.com/v1  kind: KongPlugin
  metadata:
    name: jwt-auth
    namespace: production-app
  plugin: jwt
Apply to the API routes:
SOURCE CODE / MANIFEST
  metadata:
    annotations:
      konghq.com/plugins: jwt-auth,rate-limit-backend
--- 16.4 AWS API Gateway (alternative) ---
Use AWS API Gateway v2 (HTTP  API) instead of Kong when:
SOURCE CODE / MANIFEST
  - You are fully AWS-native and want serverless routing.
  - You need AWS WAF integration.
  - You want per-endpoint throttling without running another pod.
Terraform for AWS HTTP  API Gateway:
SOURCE CODE / MANIFEST
  resource "aws_apigatewayv2_api" "main" {
    name          = "${var.project}-http-api"
    protocol_type = "HTTP"
    cors_configuration {      allow_origins = ["https://yourdomain.com"]
      allow_methods = ["GET", "POST", "PUT", "DELETE", "OPTIONS"]
      allow_headers = ["Authorization", "Content-Type"]
    }  }
  resource "aws_apigatewayv2_integration" "backend" {
    api_id             = aws_apigatewayv2_api.main.id    integration_type   = "HTTP_PROXY"
    integration_uri    = "https://yourdomain.com/api/{proxy}"
    integration_method = "ANY"
  }
  resource "aws_apigatewayv2_route" "api" {
    api_id    = aws_apigatewayv2_api.main.id
    route_key = "ANY /api/{proxy+}"
    target    = "integrations/${aws_apigatewayv2_integration.backend.id}"  }
--- 16.5 Decision: Kong vs AWS API Gateway vs Ingress-only ---
Option Best for Drawback
Nginx Ingress only Simple apps, < 10 services No rate limiting, no auth layer
Kong on K8s K8s-native, multi-service, OSS Another pod to operate
AWS API Gateway AWS-native, serverless routing Vendor lock-in, cost per call
Traefik Docker/K8s, simple config Less feature-rich than Kong
Cloudflare Workers Edge routing, global CDN Complex for stateful backends
For a Django backend at Zomato-scale: Kong + Redis rate limiting is the
standard production choice. AWS API Gateway is correct if you are routing to
Lambda or AWS services primarily.
