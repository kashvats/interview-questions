Deployment Configuration
The Deployment is the most important resource. Key production settings:
  replicas: 3 — baseline for high availability
  RollingUpdate with maxUnavailable: 0 and maxSurge: 1 — aims for zero-downtime deploys
  revisionHistoryLimit: 5 — retains rollback capability for the last 5 versions
  podAntiAffinity — spreads replicas across different nodes (weight 100)
  terminationGracePeriodSeconds: 30 — allows in-flight requests to complete
  preStop sleep 10 — drains traffic before SIGTERM is sent

CHAPTER 02
Terraform AWS Infrastructure
Terraform provisions the platform — the cloud resources your application runs on. It should never contain
application code, only infrastructure declarations.
What This Setup Provisions
  VPC with public and private subnets across multiple availability zones
  NAT gateway for private subnet internet egress
  Security groups: ALB (public 80/443) and App (ALB → 8080 only)
  Application Load Balancer with health-checked target group
  EC2 instance in private subnet with bootstrap user-data
  ALB listener forwarding HTTP → target group
Terraform Module Structure
A well-organised Terraform project separates concerns into reusable modules:

What Terraform Should NOT Manage
■  Terraform provisions the platform. Application deployment (image tags, replica counts, config values)
belongs in Kubernetes manifests or Helm charts — not in Terraform.
  Do not manage Kubernetes Deployments from Terraform (use kubectl / Helm / ArgoCD)
  Do not hardcode secrets in .tf files — use AWS Secrets Manager or Vault
  Do not run migrations from Terraform — run them as a Kubernetes Job

CHAPTER 03
Production Django with Docker
A production Django setup is not just adding Gunicorn. It requires careful layering of the WSGI server,
reverse proxy, static/media handling, security settings, and health endpoints.
What This Production Setup Includes
  Dockerfile — multi-stage build, non-root user, minimal runtime image
  entrypoint.sh — waits for DB, runs collectstatic and migrate, then starts server
  gunicorn.conf.py — CPU-aware worker count, shared-memory tmp dir
  nginx/default.conf — static/media served directly; Django proxied via upstream
  docker-compose.yml — db + web + nginx with shared volumes and health checks
  .dockerignore — excludes venv, .env, sqlite, media, test artifacts from build context
Dockerfile: Why Multi-Stage?
The multi-stage build keeps the final production image clean. The builder stage installs all dependencies;
the runtime stage copies only what is needed to run.
FROM python:3.12-slim AS base
# Install build tools, libpq-dev, curl...
FROM base AS builder
COPY requirements.txt .
RUN pip install -r requirements.txt
FROM python:3.12-slim AS runtime # Fresh clean image
COPY --from=builder /usr/local /usr/local # Only packages
COPY . .
USER appuser # Never run as root
CMD ["gunicorn", "-c", "gunicorn.conf.py", "config.wsgi:application"]

Migrations: The Mature Pattern
Running migrations inside every app container startup works for small teams but creates race conditions at
scale. The mature approach:
  One dedicated Kubernetes Job runs manage.py migrate before deployment
  App containers start Gunicorn — no migration logic
  Optional separate Job runs collectstatic
  This pattern avoids multiple pods racing to migrate simultaneously

CHAPTER 04
CI/CD Pipelines
A production CI/CD pipeline enforces quality gates before any image reaches a registry. Tests, checks,
and security scans must pass before Docker build runs — and build must pass before deployment is
triggered.
Pipeline Stages — Both Jenkins and GitHub Actions
Stage
What It Does
Checkout
Pull source code from repository
Set up Python
Create virtualenv, install dependencies
Django checks
python manage.py check — catches config errors early
Tests
Full test suite — fails fast before Docker build
Build image
docker build with versioned + latest tags
Push image
Authenticate to GHCR and push both tags
Cleanup
Prune dangling images, remove local venv
Jenkins Pipeline Key Concepts
  agent any — runs on any available Jenkins worker
  timestamps() — all log lines prefixed with timestamp
  disableConcurrentBuilds() — prevents overlapping runs for same job
  buildDiscarder(logRotator: 20) — keeps only last 20 builds
  IMAGE_TAG = BUILD_NUMBER — each build gets a unique, traceable tag
  withCredentials — registry token loaded from Jenkins secure store, never in code

GitHub Actions: Stronger Pipeline (Recommended)
Split into two jobs: quality first, then docker build (only if quality passes):
  quality job: lint (ruff), Django checks, tests, pip-audit, bandit source scan
  docker job: needs: quality — only runs if all quality gates pass
  Conditional push: image only published on push events, not on PRs
  Cache: type=gha with mode=max — significantly speeds up repeated builds
Jenkins vs. GitHub Actions
Use GitHub Actions when:
Use Jenkins when:
 Code is already on GitHub
 Company already runs Jenkins
 Want simpler setup
 Complex custom pipeline requirements
 No complex internal pipelines
 Heavy internal / private infra integration
 Modern team, greenfield project
 Existing Jenkins investment
■  For most modern GitHub-hosted projects, GitHub Actions is the cleaner default. It requires zero server
infrastructure, has first-class Docker support, and integrates natively with GHCR using GITHUB_TOKEN.
Deployment Stage: After CI Passes
Typical deployment targets and their update commands:

# Kubernetes (direct)

# Kubernetes via Helm

Final Production Blueprint
This is the architecture to deploy for any serious production project:
React → Nginx container → Kubernetes Deployment
Django → Gunicorn → Kubernetes Deployment
Celery → Worker process → Kubernetes Deployment
Celery Beat → Scheduler process → Kubernetes Deployment (replicas: 1)
Redis → Managed / dedicated → External service
Postgres → Managed (RDS/Cloud SQL) → External service
Media/Static → S3 + CDN → Object storage
Terraform → Provisions all infra → IaC
GitHub Actions → Builds & deploys → CI/CD
Ingress → Public entrypoint → NGINX Ingress Controller
Observability → Sentry + Prometheus + centralised logs
