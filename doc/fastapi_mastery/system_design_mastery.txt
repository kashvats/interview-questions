Topic: TCP level vs Application level, Envoy/Nginx logic
Project: Zomato-Global Ingress Gateway -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, Load Balancer (LB) sirf traffic distribute nahi karta, ye decide karta hai traffic "kaise" jayega.

- **Layer 4 (L4)**: Sirf IP aur Port dekhta hai (TCP/UDP). Tezi se traffic bhejta hai par use nahi pata ki request mein "Pizza" manga hai ya "Burger".

 - *Example*: Agar URL `/payment` hai toh Payment Service ko bhejo. Agar `/search` toh Search Service ko.
- **Staff Tip**: L7 flexibility deta hai (Canary/Blue-Green), par L4 high-performance background traffic ke liye best hai. -- MODULE 03: CACHING STRATEGIES - THE DATA SHORTCUT
Topic: Write-through, Write-back, Write-around
Project: Zomato-Global Menu Cache -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, DB is slow, Cache is fast. Par data update kaise karein?

- **Write-through**: Data pehle Cache mein likho, fir DB mein. (Most consistent, but write latency high).
- **Write-around**: Data seedha DB mein likho, Cache skip kardo. (Best for data not immediately needed).
- **Write-back**: Data sirf Cache mein likho, DB mein baad mein (asynchronously). (Sabse fast, par data loss ka khatra!).
- **Staff Rule**: Zomato Menu ke liye Write-through best hai taaki "Sold Out" status turant dikhe. -- MODULE 04: CACHE INVALIDATION - THE HARDEST PROBLEM

2. LOGIC: CONSISTENT HASHING SKELETON (Python-ish) -- ```python
import hashlib

class ConsistentHashing:
 def __init__(self, nodes, replicas=3):
 self.replicas = replicas
 self.ring = {}
 for node in nodes:
 self.add_node(node)

 def add_node(self, node):
 for i in range(self.replicas):
 key = self.get_hash(f"{node}:{i}")
 self.ring[key] = node
 self.ring = dict(sorted(self.ring.items()))

 def get_node(self, key):
 h = self.get_hash(key)
 for ring_key in self.ring:
 if h <= ring_key:
 return self.ring[ring_key]
 return list(self.ring.values())[0]

 def get_hash(self, key):
 return int(hashlib.md5(key.encode()).hexdigest(), 16)

# Usage: ch.get_node("user_123_session")
```

================================================================================ -- MODULE 11: CAP THEOREM - THE IRON TRIANGLE
Topic: Consistency, Availability, Partition Tolerance
Project: Zomato-Global Distributed Choice -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, Distributed Systems mein aap sab kuch ek sath nahi paa sakte.

- **Consistency (C)**: Sabhi users ko same time par same data mile.
- **Availability (A)**: Server hamesha response de (chahe data purana ho).
- **Partition Tolerance (P)**: Agar network breakdown ho jaye (partitions), toh bhi system chalta rahe.
- **The Rule**: P hamesha zaruri hai distributed system mein. Isliye aapko choose karna hoga: **CP** (Consistency) ya **AP** (Availability).
- **Staff Value**: Zomato Checkout mein **CP** chahiye (Double booking nahi honi chahiye). Discovery mein **AP** chalta hai. -- MODULE 12: PACELC THEOREM - BEYOND CAP
Topic: Latency vs Consistency, The real-world extension
Project: Zomato-Global Performance Tradeoff -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, CAP theorem sirf "Network Failure" ki baat karta hai. Par normal time par kya?

- **PACELC**: If **P** (Partition), choice is **A** or **C**. **E**lse (Normal), choice is **L**atency or **C**onsistency.
- **The Lesson**: Agar system healthy hai, toh kya aap faster response (Latency) chahte hain ya 100% accurate data (Consistency)?
- **Staff Choice**: Cassandra/DynamoDB PACELC follow karte hain. Inme aap "Tunable Consistency" set kar sakte hain. -- MODULE 13: DISTRIBUTED TRANSACTIONS - 2PC
Topic: Two-Phase Commit, The Coordinator, Blocking problem
Project: Zomato-Global Inventory Sync -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, agar 2 alag DBs (e.g., Inventory and Payments) ko update karna hai, toh "All or Nothing" kaise hoga?

- **2PC**:
 1. **Prepare Phase**: Coordinator sabko puchta hai: "Tayyar ho?"
 2. **Commit Phase**: Agar sab "Yes" bole, toh commit. Agar ek bhi "No", toh abort.
- **The Flaw**: Ye synchronous aur blocking hai. Agar Coordinator fail ho jaye, system "Deadlock" mein ja sakta hai. Staff engineers use usually avoid karte hain high-scale par. -- MODULE 14: THE SAGA PATTERN - COMPENSATING TRANSACTIONS

Project: Zomato-Global Order Workflow -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, 2PC ke fail hone ke baad **Saga** aaya.

- **Concept**: Ek badi transaction ko choti sequential transactions mein todna.
- **Choreography**: Har service event emit karti hai, doosri use listen karti hai. (Decoupled).
- **Compensating Action**: Agar Payments fail ho gaye, toh "Refund" ya "Order Cancel" ka naya event chalao pehle waale changes undo karne ke liye.

- **Consensus**: Sabhi nodes ek value par agree karte hain.
- **Raft**: Ye asan hai samajhne mein.
