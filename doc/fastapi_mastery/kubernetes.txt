================================================================================ -- MODULE 01: CONTROL PLANE LIFECYCLE - THE BRAIN OF THE CLUSTER
Topic: etcd, API Server, Scheduler, and Controller Manager
Project: Zomato-Infra Core Setup -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, Kubernetes koi ek "Software" nahi hai, ye distributed systems ka ek cluster hai. Iska sabse important part hai **Control Plane**.

- **etcd**: Ye cluster ka "Single source of truth" hai. Isme sara data (Deployment status, Secrets, Configs) Key-Value pair mein store hota hai. Ye **Raft Consensus** algorithm use karta hai-iska matlab hai ki agar nodes fail ho jayein, tab bhi data consistent rahega.
- **kube-apiserver**: Ye cluster ka entry point hai. Har request (kubectl ya internal) pehle yahan aati hai. Ye "Gatekeeper" hai jo authentication aur validation karta hai.
- **kube-scheduler**: Iska kaam hai naye Pods ke liye sahi Node dhundhna. Ye "Resource availability" aur "Taints/Tolerations" check karta hai.
- **kube-controller-manager**: Ye "Watchdog" hai jo hamesha check karta hai ki "Desired State" aur "Actual State" match kar rahe hain ya nahi.

**Staff Level Tip**: Production mein etcd hamesha odd number of nodes (3, 5, 7) par hona chahiye taaki quorum maintain ho sake. -- MODULE 02: KUBELET & RUNTIME - THE WORKER'S ARM
Topic: CRI (Container Runtime Interface), Containerd, and OCI
Project: Zomato-Infra Node Lifecycle -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, har Worker Node par ek agent chalta hai jise **kubelet** kehte hain.

- **Kubelet**: Ye API server se instructions leta hai aur Node par containers ko manage karta hai. Ye "Node health" ki report wapas API server ko deta hai.
- **CRI (Container Runtime Interface)**: Pehle K8s sirf Docker samajhta tha, par ab wo CRI use karta hai. Isse hum `containerd` ya `CRIO` jaise lightweight runtimes use kar sakte hain.
- **OCI (Open Container Initiative)**: Ye containers ka standard hai. Koi bhi image jo OCI compliant hai, wo K8s par chal jayegi.

**Staff Perspective**: Docker ab K8s mein mandatory nahi hai. Production mein hum primarily `containerd` use karte hain kyunki wo faster aur resource-efficient hai. -- MODULE 03: DECLARATIVE STATE - THE YAML PHILOSOPHY
Topic: Spec vs Status, Level-Triggering, and Resource Versions
Project: Zomato-Infra Manifest Design -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, K8s mein hum "How" (kaise karna hai) nahi bolte, hum "What" (kya chahiye) bolte hain. Is ise **Declarative Management** kehte hain.

- **Spec**: Hamari "Desired State" (Mujhe 3 replicas chahiye).
- **Status**: Cluster ki "Actual State" (Abhi 2 replicas chal rahe hain).
- **Control Loop**: K8s hamesha Status ko Spec ke barabar lane ki koshish karta hai.
- **ResourceVersion**: Har object ka ek version hota hai. Isse "Optimistic Concurrency" maintain hoti hai-do log ek sath object change nahi kar sakte bina version check kiye. -- MODULE 04: NAMESPACES & MULTI-TENANCY
Topic: ResourceQuotas, LimitRanges, and Isolation
Project: Zomato-Infra Internal Boundaries (Dev/Staging/Prod) -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, ek cluster mein multiple teams aur projects ko manage karne ke liye hum **Namespaces** use karte hain.

- **Namespace**: Ye ek "Virtual Cluster" hai. Ek namespace ke andhar resources ek doosre ko easily find kar sakte hain.
- **ResourceQuota**: Ye staff level engineering ke liye mandatory hai. Ye fix karta hai ki ek namespace kitni CPU/RAM consume kar sakta hai.
- **LimitRanges**: Ye har Pod ke liye "Minimum" aur "Maximum" resource limits set karta hai taaki koi "Greedy Pod" pure node ke resources na kha jaye.

**Pro Tip**: Kabhi bhi `default` namespace production ke liye use na karein. Hamesha structured namespaces (`zomato-prod-api`, `zomato-staging-db`) banayein. -- MODULE 05: POD ANATOMY & THE LIFECYCLE
Topic: Init Containers, Sidecars, and Lifecycle Hooks
Project: Zomato-Infra Application Pods -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, Pod K8s ka sabse chota unit hai. Ye sirf ek container nahi, balki multiple containers ka group ho sakta hai jo shared storage aur network share karte hain.

- **Init Containers**: Ye main container se pehle chalte hain. Inka kaam hai "Preparation" karna (Jaise DB migrations ya configuration fetch karna).
- **Sidecar Containers**: Ye main container ke sath chalte hain (Jaise logging agent ya service mesh proxy).
- **Lifecycle Hooks**:
 - `PostStart`: Container start hote hi execute hota hai.
 - `PreStop`: Container delete hone se theek pehle execute hota hai (Graceful shutdown ke liye essential).

2. CODE: THE ANATOMY OF A PRODUCTION POD -- ```yaml
apiVersion: v1
kind: Pod
metadata:
 name: zomato-api-pod
 labels:
 app: zomato-api
spec:
 initContainers:
 - name: wait-for-db
 image: busybox
 command: ['sh', '-c', 'until nc -z db-service 5432; do sleep 2; done;']
 containers:
 - name: main-api
 image: zomato/api:v1.2
 resources:
 limits:
 cpu: "500m"
 memory: "512Mi"
 requests:
 cpu: "200m"
 memory: "256Mi"
 lifecycle:
 preStop:
 exec:
 command: ["/bin/sh", "-c", "sleep 15"] # Give time to drain traffic
 - name: logger-sidecar
 image: fluentd:latest
```

7. STAFF-LEVEL PRO TIPS (PART 1) -- - **Kill your darlings**: Pods ephemeral hote hain. Kabhi bhi kisi Pod ke stateful hone par bharosa na karein (Unless it's a StatefulSet).
- **Strict Limits**: Hamesha resource limits define karein. Bina limits ke Pods, cluster stability ke liye cancer hain.
- **Metadata Matters**: Annotations aur Labels ko religiously use karein for tracing and cost allocation.

================================================================================ -- MODULE 06: ADVANCED SCHEDULING - TAINTS, TOLERATIONS & AFFINITY
Topic: Node Isolation, Workload Placement, and Anti-affinity
Project: Zomato-Infra Multi-Zone Placement -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, har Pod har Node par nahi chalna chahiye. Humein "Place" aur "Constraint" ka balance banana padta hai.

- **Taints**: Ye Node par "Daag" ki tarah hain jisse Pods us node se "Repel" hote hain. (e.g., `gpu=true:NoSchedule`).
- **Tolerations**: Ye Pod par wo "Davai" hai jo use tainted node par chalne ki permission deti hai.
- **Node Affinity**: Ye "Preference" hai. (e.g., Ye Pod sirf SSD nodes par hi chalna chahiye).
- **Inter-pod Anti-affinity**: Staff level practice. Ye ensures karta hai ki ek hi app ke do replicas ek hi node ya ek hi rack par na ho (High Availability ke liye). -- MODULE 07: PRIORITY & PREEMPTION
Topic: PriorityClasses, Resource Starvation, and Eviction
Project: Zomato-Infra Critical Checkout Service -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, jab cluster full ho jaye, toh humein decide karna padta hai ki kaunsa Pod system ke liye "VVIP" hai.

- **PriorityClass**: Hum Pods ko priority score dete hain (e.g., System-critical: 1000000, Batch-job: 100).
- **Preemption**: Agar ek high-priority Pod ko jagah chahiye aur cluster full hai, toh K8s lower priority Pods ko "Kick out" (Preempt) kar deta hai taaki VIP Pod chal sake.
- **Eviction**: Jab Node ka memory/disk khatam ho jaye, toh Kubelet automatically Pods ko terminate karna shuru karta hai (Critical ones are saved last). -- MODULE 08: DEPLOYMENT STRATEGIES
Topic: RollingUpdate, MaxSurge, MaxUnavailable, and Canary Pattern
Project: Zomato-Infra Zero-Downtime Releases -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, naya code deploy karte waqt "Zero Downtime" hamesha goal hona chahiye.

- **RollingUpdate**: Purane Pods ko ek-ek karke naye se replace karna.
 - `maxSurge`: Update ke waqt kitne extra Pods upar ja sakte hain.
 - `maxUnavailable`: Update ke waqt kitne Pods down ho sakte hain.
- **Canary Release**: Naya version sirf 5% users ko dikhana. K8s mein ye hum labels aur dedicated service manipulation se karte hain.
- **Recreate**: Purane saare band karke naye chalana (Only for non-critical tools). -- MODULE 09: STATEFULSETS - THE DATABASE MANAGER
Topic: Stable Network Identity, Headless Services, and Ordinal Index
Project: Zomato-Infra Redis Cluster -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, Databases ko "Identity" chahiye hoti hai. Agar ek DB replica restart ho, toh use vahi "Disk" aur vahi "Hostname" wapas milna chahiye.

- **Stable ID**: StatefulSet mein Pods ke naam `mysql-0`, `mysql-1` hote hain (Deployments ki tarah random hash nahi).
- **Headless Service**: Ye IP forwarding ki jagah direct Pod IPs ka DNS record deta hai taaki replication set-up ho sake.
- **VolumeClaimTemplates**: Har Pod ko apna dedicated storage (PersistentVolume) milta hai. -- MODULE 10: BATCH & CRON JOBS
Topic: Completion Strategy, Parallelism, and BackoffLimits
Project: Zomato-Infra Nightly Audit & Cleanup -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, har cheez server ki tarah forever nahi chalti. Kuch "One-time" tasks hote hain.

- **Jobs**: Ek task jo execute hoga aur finish ho jayega. (e.g., Database Migrations).
- **CronJobs**: Ek periodic task. (e.g., Har raat 2 AM par report generate karna).
- **RestartPolicy**: Jobs mein hamesha `OnFailure` ya `Never` hona chahiye, `Always` kabhi nahi (Warna Job loop mein chalta rahega).

2. CODE: THE ARCHITECT'S CRONJOB -- ```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
 name: zomato-nightly-backup
spec:
 schedule: "0 2 * * *" # Every night at 2 AM
 jobTemplate:
 spec:
 template:
 spec:
 containers:
 - name: backup-tool
 image: zomato/backup:latest
 restartPolicy: OnFailure
 backoffLimit: 4
```

7. STAFF-LEVEL PRO TIPS (PART 2) -- - **Pod Disruption Budgets (PDB)**: Hamesha PDB banayein taaki cluster maintenance ke waqt aapke critical Pods minimum count se neeche na jayenge.
- **Topology Spread Constraints**: Sirf node anti-affinity kaafi nahi hai. Use `topologySpreadConstraints` taaki Pods equal ratio mein saare Availability Zones (AZ) mein spread hon.
- **HPA Tuning**: Horizontal Pod Autoscaler set karte waqt memory aur CPU ke metric thresholds ko conservative rakhein taaki "Flapping" na ho.

- **CoreDNS**: Ye cluster ka "Yellow Pages" hai. Jab hum ek service banate hain `zomato-service`, CoreDNS uska ek entry `zomato-service.namespace.svc.cluster.local` bana deta hai.

Topic: ClusterIP, NodePort, LoadBalancer, and Headless Services
Project: Zomato-Infra Traffic Exposure -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, service ko "Inside" ya "Outside" expose karne ke 4 main raste hain.

- **ClusterIP**: Sirf cluster ke andhar access (Default).
- **NodePort**: Har worker node par ek dedicated port open karna. (Don't use for prod unless necessary).

Project: Zomato-Infra Advanced Routing (v2 Beta) -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, **Ingress** ab legacy bans chuka hai. Hum 2026 mein **Gateway API** use karte hain.

- **Gateway**: Ye infrastructure load-balancer define karta hai (Admin manage karta hai).
- **HTTPRoute**: Ye routing logic (Paths, Headers) define karta hai (Developer manage karta hai).
- **Separation of Concerns**: Isse Infrastructure teams aur App teams alag-alag manage kar sakti hain bina ek doosre ke system ko break kiye. -- MODULE 14: NETWORK POLICIES - ZERO TRUST
Topic: Pod-to-Pod isolation, Ingress/Egress rules, and IPBlock
Project: Zomato-Infra Payment Gateway Security -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, default mein K8s mein "Saare Pods saare Pods se baat kar sakte hain." Ye security suicide hai.

- **Zero Trust**: Humein "Default Deny" policy lagani chahiye.
- **NetworkPolicy**: Hum YAML mein define karte hain ki `payment-pod` sirf `db-pod` se baat kare aur kisi se nahi.
- **Labels are everything**: Policies Pod names par nahi, humesha labels par lagti hain. -- MODULE 15: SERVICE MESH - ISTIO vs LINKERD
Topic: mTLS, Observability, and Circuit Breaker
Project: Zomato-Infra Global Mesh Audit -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, jab cluster mein 100+ microservices hon, toh network management "Chain-of-command" ban jata hai.

- **mTLS**: Har connection ko encrypt karna (Service A to Service B).
- **Circuit Breaker**: Agar Service B slow hai, toh Service A request bhejana band karade taaki "Cascading failure" na ho.
- **Staff-Level Advice**: Pehle standard K8s networking simple rakhein. Service Mesh tabhi lagayein jab mTLS ya complex traffic splitting (A/B testing) mandatory ho. Linkerd for simplicity, Istio for full-blown features.

2. CODE: GATEWAY API MANIFEST (HTTPROUTE) -- ```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
 name: zomato-api-route
spec:
 parentRefs:
 - name: prod-gateway
 rules:
 - matches:
 - path:
 type: PathPrefix
 value: /api/v1
 backendRefs:
 - name: zomato-v1-service
 port: 80
 weight: 90
 - name: zomato-v2-canary
 port: 80
 weight: 10 # 10% Traffic to Canary
```

7. STAFF-LEVEL PRO TIPS (PART 3) -- - **Avoid NodePort in Prod**: Ye cluster ports ko expose karta hai, security risk hai. Hamesha Ingress/Gateway API + LoadBalancer use karein.
- **DNS caching**: CoreDNS par workload badhne na dein. `NodeLocal DNSCache` install karein taaki DNS lookups faster ho sakein.
- **Default Deny Policy**: Har naye namespace mein hamesha ek "Deny-All" NetworkPolicy lagayein aur fir ek-ek karke raste kholein.

================================================================================ -- MODULE 16: PV/PVC LIFECYCLE - BINDING & RECLAIM
Topic: PersistentVolumes, Claims, and Bound States
Project: Zomato-Infra Database Storage -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, Pods temporary hote hain, par data permanent hona chahiye. K8s mein storage manage karne ke liye hum teen concepts use karte hain:

- **PersistentVolume (PV)**: Ye "Real Storage" hai (Jaise AWS EBS volume ya physical hard disk). Iska specification admin karta hai.
- **PersistentVolumeClaim (PVC)**: Ye user ki "Request" hai. Pod directly disk nahi maangta, wo PVC mangta hai. ("Mujhe 10GB ReadWriteOnce storage chahiye").
- **Binding**: Jab K8s ek sahi PV dhoondh kar use PVC se link kar deta hai, use "Bound" state kehte hain.
- **Reclaim Policy**: Jab PVC delete hota hai, toh PV ka kya hoga? 
 - `Retain`: Manual cleaning.
 - `Delete`: Storage bhi udd jayega (Careful!). -- MODULE 17: STORAGECLASSES - DYNAMIC PROVISIONING
Topic: Provisioners, Binding Modes, and Volume Expansion
Project: Zomato-Infra On-demand Disks -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, har baar admin manual PV banaye, ye scale par possible nahi hai.

- **Dynamic Provisioning**: Hum hamesha **StorageClass** use karte hain. Jab PVC aata hai, K8s automatically cloud (AWS/Azure) ko bolta hai "Ek naya disk banao aur attach kardo."
- **Binding Mode**:
 - `Immediate`: PVC bante hi disk ban jayega.
 - `WaitForFirstConsumer`: Disk tabhi banega jab Pod actually start hoga. Ye multi-AZ scheduling ke liye important hai.
- **Volume Expansion**: Agar app ka data badh jaye, toh hum PVC ka size update karke disk live expand kar sakte hain (if SC supports it). -- MODULE 18: CSI INTERNALS - THE STORAGE PLUG
Topic: Container Storage Interface, NodePublish, and External Provisioners
Project: Zomato-Infra Storage Driver Audit -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, pehle storage code K8s ke core binary mein hota tha (In-tree). Ab hum **CSI (Container Storage Interface)** use karte hain.

- **Storage Agnostic**: CSI se developers ko K8s update karne ki zarurat nahi agar unhe naya storage support chahiye (Jaise NetApp ya PureStorage).
- **NodePublish**: Ye wo step hai jahan CSI driver actually disk ko worker node par "mount" karta hai taaki container use dekh sake. -- MODULE 19: SNAPSHOTTING & DISASTER RECOVERY
Topic: VolumeSnapshots, Velero, and RPO/RTO
Project: Zomato-Infra Backup Strategy -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, Staff level engineer hamesha "Worst case" ke liye ready rehta hai.

- **VolumeSnapshot**: Ye disk ka ek "Point-in-time" photo lena hai. Agar DB corrupt ho jaye, toh hum naya disk isi snapshot se bana sakte hain.
- **Velero**: K8s backup ka standard tool. Ye aapke YAML states aur actual storage data dono ko S3 bucket mein backup karta hai.
- **RPO/RTO**: Recovery Point (kitna data lose kar sakte hain) aur Recovery Time (kitni jaldi system wapas aayega). -- MODULE 20: LOCAL PERSISTENT VOLUMES (HIGH IOPS)
Topic: Local NVMe, Node Affinity, and DB Performance
Project: Zomato-Infra Low-Latency Cache -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, kabhi kabhi cloud network storage (EBS) slow hota hai (IOPS limits). Databases ko "Phadu performance" chahiye hoti hai.

- **Local PV**: Worker node ki physical SSD/NVMe ko use karna.
- **Trade-off**: Agar node delete ho gaya, toh data lost! Isliye hum ise sirf un replicats/databases ke liye use karte hain jo application layer par replication handle karte hain (Jaise Cassandra ya ElasticSearch).
- **Node Affinity Mandatory**: Local PV specific node se "Chipka" hota hai, toh Pod hamesha usi node par schedule hona chahiye.

2. CODE: STORAGECLASS & PVC WITH WAITFORFIRSTCONSUMER -- ```yaml
# 1. StorageClass
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
 name: zomato-ssd-sc
provisioner: ebs.csi.aws.com
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer # Important for Multi-AZ
allowVolumeExpansion: true

---
# 2. PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
 name: zomato-db-pvc
spec:
 accessModes:
 - ReadWriteOnce
 resources:
 requests:
 storage: 50Gi
 storageClassName: zomato-ssd-sc
```

7. STAFF-LEVEL PRO TIPS (PART 4) -- - **Access Modes**: Hamesha check karein `ReadWriteOnce` (ek hi node par mount) vs `ReadWriteMany` (multiple nodes, jaise NFS). Prod DBs hamesha RWO hote hain safety ke liye.
- **Garbage Collection**: Reclaim policy ko hamesha Prod mein `Retain` rakhein agar data manual safety chahiye, warna nikal jayega.
- **Storage Monitoring**: HPA storage par kaam nahi karta. Disk pressure monitor karein (Kube-state-metrics) taaki Pod crash hone se pehle alerts aa jayein.

- **ServiceAccount**: Pods ko cluster se baat karne ke liye ek identity chahiye hoti hai.

Topic: Mutating vs Validating Webhooks
Project: Zomato-Infra Policy Enforcement -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, API server par request jaane se pehle hum use intercept kar sakte hain.

- **Validating Webhook**: Request ko "Review" karna. Agar Pod mein `labels` missing hain, toh request reject kardo.
- **Mutating Webhook**: Request ko "Change" karna. Agar Pod mein sidecar container missing hai, toh use automatically add kardo (Jaise Istio karta hai).

Topic: Privileged, Baseline, and Restricted Profiles
Project: Zomato-Infra Pod Hardening -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, `PodSecurityPolicy` (PSP) ab khatam ho chuka hai. Ab hum **Pod Security Admission** use karte hain.

- **Privileged**: Sab kuch allowed (Only for system pods).
- **Baseline**: Common sense security (No hostPath, etc.).
- **Restricted**: Sabse secure profile. Pod root privilege nahi le sakta, host network use nahi kar sakta.
- **Enforcement**: Hum namespace par labels lagate hain: `pod-security.kubernetes.io/enforce: restricted`. -- MODULE 24: SECRETS MANAGEMENT & EXTERNAL SECRETS

- **Sealed Secrets**: Secrets ko encrypt karke Git mein store karna. Sirf cluster use decrypt kar sakta hai.
- **Etcd Encryption**: Cloud level par etcd disk ko encrypt karna mandatory hai. -- MODULE 25: NETWORK HARDENING & CNI ENCRYPTION
Topic: WireGuard, IPsec, and CNI Identity
Project: Zomato-Infra Encrypted Transit -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, kya pods ke beech ka traffic secure hai? 

- **CNI Encryption**: Cilium ya Calico jaise modern CNIs **WireGuard** ya **IPsec** use karke Pod-to-Pod traffic ko automatically encrypt karte hain.
- **mTLS vs CNI Encryption**: CNI encryption layer 3 par hoti hai, jabki Service Mesh (mTLS) layer 7 par. High security environments mein dono use karte hain.

2. CODE: RBAC ROLE & BINDING FOR LOGS -- ```yaml
# 1. Define Role
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
 namespace: zomato-prod
 name: log-reader
rules:
- apiGroups: [""]
 resources: ["pods", "pods/log"]
 verbs: ["get", "list", "watch"]

---
# 2. Bind to ServiceAccount
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
 name: read-logs-binding
 namespace: zomato-prod
subjects:
- kind: ServiceAccount
 name: monitor-sa
roleRef:
 kind: Role
 name: log-reader
 apiGroup: rbac.authorization.k8s.io
```

- **Zero-Trust Network**: NetworkPolicy mein hamesha `ipBlock` use karein external traffic restrict karne ke liye.
