================================================================================ -- MODULE 01: THE BUFFER POOL & SHARED BUFFERS
Topic: Memory Caching, Hit Ratios, and OS Page Cache
Project: Zomato-Global Cache Audit -- 1. PROD-GRADE LOGIC (PSQL) -- Check your Buffer Cache Hit Ratio (Should be > 99%)
-- If this is low, your 'shared_buffers' is insufficient for your warm data set.
SELECT 
 sum(heap_blks_read) as disk_read,
 sum(heap_blks_hit) as buffer_hit,
 round(sum(heap_blks_hit) / (sum(heap_blks_hit) + sum(heap_blks_read) + 0.0001) * 100, 2) as hit_ratio_percentage
FROM pg_statio_user_tables;

-- Inspect which tables are eating up your Shared Buffers
-- Requires: CREATE EXTENSION pg_buffercache;
SELECT 
 c.relname, 
 count(*) AS buffers
FROM pg_buffercache b
INNER JOIN pg_class c ON b.relfilenode = pg_relation_filenode(c.oid)
AND b.reldatabase IN (0, (SELECT oid FROM pg_database WHERE datname = current_database()))
GROUP BY c.relname
ORDER BY buffers DESC
LIMIT 10;

2. CONCEPT EXPLANATION (Hinglish) -- Dosto, DB speed ka raaz RAM mein hai. 

- **Shared Buffers**: Ye wo memory hai jahan Postgres "Data Pages" ko cache karta hai.
- **Hit Ratio**: Agar ye 99% hai, matlab 100 mein se 99 baar data RAM se mil gaya. Agar ye kam hai, toh Postgres ko disk (slow) par jana pad raha hai.
- **Staff Value**: Hamesha `shared_buffers` ko Total RAM ka 25% set karein. Baaki RAM OS ke "Page Cache" ke liye chhodein, kyunki Postgres OS cache par bhi depend karta hai (Double Buffering). -- MODULE 02: B-TREE AUDIT & BLOAT DETECTION
Topic: Index Fragmentation, Leaf page splits, and Reindexing
Project: Zomato-Global Index Health -- 1. PROD-GRADE LOGIC (PSQL) -- Find the most fragmented indexes that need a vacuum or reindex
-- Requires: CREATE EXTENSION pgstattuple;
SELECT

 now() - last_vacuum as last_vacuum_age,
 avg_leaf_density, -- If < 70%, index is fragmented

FROM pg_stat_user_indexes i
JOIN pgstattuple(i.indexrelid::regclass::text) ON true

ORDER BY avg_leaf_density ASC;

-- The Fix: Zero-Downtime Reindexing
-- REINDEX INDEX CONCURRENTLY idx_users_email;

Topic: pg_locks, AccessExclusiveLock, and RowLock
Project: Zomato-Global Transaction Safety -- 1. PROD-GRADE LOGIC (PSQL) -- Find queries that are currently blocked by other queries
SELECT
 blocked_locks.pid AS blocked_pid,
 blocking_locks.pid AS blocking_pid,
 blocked_activity.query AS blocked_statement,
 blocking_activity.query AS blocking_statement
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_locks.pid = blocked_activity.pid
JOIN pg_catalog.pg_locks blocking_locks 
 ON blocking_locks.locktype = blocked_locks.locktype
 AND blocking_locks.DATABASE IS NOT DISTINCT FROM blocked_locks.DATABASE
 AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
 AND blocking_locks.page IS NOT DISTINCT FROM blocked_locks.page
 AND blocking_locks.tuple IS NOT DISTINCT FROM blocked_locks.tuple
 AND blocking_locks.virtualxid IS NOT DISTINCT FROM blocked_locks.virtualxid
 AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
 AND blocking_locks.classid IS NOT DISTINCT FROM blocked_locks.classid
 AND blocking_locks.objid IS NOT DISTINCT FROM blocked_locks.objid
 AND blocking_locks.objsubid IS NOT DISTINCT FROM blocked_locks.objsubid
 AND blocking_locks.pid != blocked_locks.pid
 JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid
 WHERE NOT blocked_locks.GRANTED;

2. CONCEPT EXPLANATION (Hinglish) -- Dosto, "Locks" distributed systems ka sachaayi hain par excessive locks "Deadlock" ya latency badhate hain.

- **Staff Strategy**: Transactions ko hamesha Chota (Small) rakhein. Request ke beech mein "User Input" ka wait kabhi na karein (Hold locks). -- MODULE 08: GIN & GIST INDEX PERFORMANCE
