================================================================================ -- MODULE 11: CANARY DEPLOYMENT - WEIGHTED TRAFFIC
Topic: weight shift, v1 vs v2, and Incremental rollout
Project: Zomato-Infra Safer Releases -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, "Canary" matlab naya version launch karna par sirf thode users ke liye (e.g., 5%).

- **VirtualService Weight**: Aap `VirtualService` mein `weight` attribute define karte hain. 
- **The Shift**: Pehle 95% traffic purane version par, 5% naye par. Agar logs sahi hain, toh slowly 10%, 20%... 100% tak shift kar do.
- **Why?**: Agar naye code mein bug hai, toh sirf 5% users hi affect honge, baki 95% safe rahenge. -- MODULE 12: BLUE-GREEN DEPLOYMENT LOGIC
Topic: Switch-over, Zero downtime, and Instant rollback
Project: Zomato-Infra Global Switch -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, Blue-Green mein hum 100% traffic ek sath shift karte hain.

- **Setup**: Blue (Live environment) aur Green (Naya environment) dono ready hote hain.
- **The Cutover**: Istio `VirtualService` ko update karo aur traffic Green par bhej do.
- **Rollback**: Agar naye version mein problem hai, toh turant YAML change karke traffic wapas Blue par bhej do. (No server restarts!). -- MODULE 13: LOAD BALANCING ALGORITHMS
Topic: Round Robin, Random, Least Conn, and Consistent Hash
Project: Zomato-Infra Smart Distribution -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, Istio sirf random traffic nahi bhejta.

- **Least Connection**: Naya request us pod ke paas jayega jo sabse kam busy hai.
- **Consistent Hash**: User ki `Session ID` dekh kar use hamesha same pod par bhejna (Sticky session replacement).
- **Staff Choice**: Web services ke liye `LEAST_CONN` hamesha best performance deta hai. -- MODULE 14: CIRCUIT BREAKING - THE FUSE
Topic: Max connections, Max pending requests, and Failure thresholds
Project: Zomato-Infra System Stability -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, imagine karo ek service slow ho gayi hai. Us par aur traffic bhejna aag mein tel dalne barabar hai.

- **Circuit Breaker**: Istio monitor karta hai. Agar ek service fail ho rahi hai, toh Istio us "Circuit" ko "Open" kar deta hai. 
- **Self-Healing**: Agli requests turant error return kar dengi bina backend ko pareshan kiye, taaki backend ko "Saans leni ki jagah" (time) mil jaye recovery ke liye. -- MODULE 15: FAULT INJECTION - CHAOS ENGINEERING
Topic: Aborting requests, Adding delay, and Testing resilience
Project: Zomato-Infra Chaos Test -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, application ki strength check karne ke liye humein purposely system ko break karna padta hai.

- **Delay Injection**: Istio ko bolo: "Har 10th request mein 5 second ka delay add karo." Dekho application handle kar paati hai ya nahi.
- **Abort Injection**: Specific errors (e.g., 500 Internal Error) simulate karo bina backend code change kiye. -- MODULE 16: OUTLIER DETECTION - PASSIVE HEALTH CHECK

Project: Zomato-Infra Auto Ejection -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, agar ek pod "Pagal" (unhealthy) ho gaya hai aur random 500 errors de raha hai, toh use hatao!
