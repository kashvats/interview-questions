================================================================================ -- MODULE 01: GHA ARCHITECTURE - WORKFLOWS, JOBS, & STEPS
Topic: Anatomy of an Action, Runners, and the execution model
Project: Zomato-Infra GHA Skeleton -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, GitHub Actions (GHA) ek **Event-Driven** system hai. Iska matlab hai ki jab GitHub par kuch hota hai (Push, PR, Star), tabhi ye trigger hota hai.

- **Workflow**: Ye aapki poori process hai (e.g., CI/CD). Ye `.github/workflows/` folder mein YAML file ke roop mein rehti hai.
- **Job**: Ek workflow mein multiple jobs ho sakti hain. By default, ye parallel chalti hain. Har job ek "Runner" machine par execute hoti hai.
- **Step**: Job ke andhar chote-chote tasks (e.g., Run a script, Checkout code). Steps hamesha sequentially (ek ke baad ek) chalte hain. -- MODULE 02: RUNNERS - HOSTED VS SELF-HOSTED
Topic: Virtual machines, OS choices, and Cost vs Security
Project: Zomato-Infra Runner Selection -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, code kahan execute ho raha hai? "Runner" machine par.

- **GitHub-Hosted Runners**: GitHub aapko Ubuntu, Windows, aur macOS machines deta hai. Privacy high hai par large builds par cost badh sakti hai.
- **Self-Hosted Runners**: Aap apna server (AWS EC2 ya personal machine) GitHub se connect kar sakte hain. 
- **Staff-Level Tip**: Organizations hamesha self-hosted runners prefer karti hain heavy builds ke liye taaki binary caching fast ho aur private VPC resources access ho sakein. -- MODULE 03: TRIGGER EVENTS - THE "ON" KEYWORD
Topic: Push, Pull_Request, Schedule (Cron), and Manual triggers
Project: Zomato-Infra Event Hub -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, GHA ko batana padta hai ki "Kab chalna hai?".

- **push**: Jab code main branch par merge ho.
- **pull_request**: Jab koi PR open kare ya update kare.
- **schedule**: Cron syntax use karke (e.g., Har raat 12 baje security scan).
- **workflow_dispatch**: Ye Staff developers ka favorite hai. Isse aap UI mein ek button dete ho taaki pipeline manually trigger ho sake variables ke sath. -- MODULE 04: YAML SYNTAX ANATOMY
Topic: Structural rules, Indentation, and Name mappings
Project: Zomato-Infra Syntax Standard -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, GHA YAML par chalta hai. Ek ghalat space aur workflow fail ho jayega.

- **Structure**: Hamesha `name`, `on`, `jobs` top-level par hote hain.
- **uses**: Jab hum pehle se bani banayi "Action" (plugin) use karte hain (e.g., `actions/checkout@v4`).
- **run**: Jab hum direct terminal command chalate hain. -- MODULE 05: ENVIRONMENT VARIABLES (ENV)
Topic: Workflow level, Job level, and Step level scopes

- **Note**: Secrets sirf repo admins hi dekh/change kar sakte hain. -- MODULE 07: JOB DEPENDENCIES (NEEDS)
Topic: Sequence, DAG (Directed Acyclic Graph), and Failure cascading
Project: Zomato-Infra Pipeline Flow -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, agar hum nahi chahte ki "Deploy" job chale agar "Test" job fail ho gayi hai, toh hum `needs` use karte hain.

Project: Zomato-Infra Dynamic Logic -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, GHA ko "Sensing" power context objects se milti hai.

- **github context**: Aapko info deta hai: "Commit kisne kiya?", "PR number kya hai?", "Repo name kya hai?".
- **steps context**: Ek step se output generate karke doosre mein use karne ke liye.

Topic: branches filters, paths filters, and ignorable paths
Project: Zomato-Infra Monorepo Routing -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, Staff level monorepo mein hum humesha `paths` filter use karte hain.

- **Use Case**: Agar kisi ne sirf `docs/` folder mein change kiya hai, toh pura CI pipeline chalane ka fayda nahi (Billing save!).
- **Syntax**: `paths: ['src/**']` - Workflow tabhi chalega jab `src` folder ke andhar koi file change hogi. -- MODULE 10: CONCURRENCY CONTROL
Topic: groups, cancel-in-progress, and Resource locking
Project: Zomato-Infra Run Optimization -- 1. CONCEPT EXPLANATION (Hinglish) -- Dosto, agar ek developer ne PR par 5 commit kiye 1 minute mein, toh 5 parallel builds chal jayenge. Ye wasteful hai.

- **concurrency group**: Ek group name do (e.g., `${{ github.ref }}`).
- **cancel-in-progress**: true - Isse naya commit aate hi purani build automatically cancel ho jayegi. Staff engineering ka golden rule for cost-saving.

2. CODE: A PRODUCTION-GRADE GHA WORKFLOW -- ```yaml
name: Zomato CI Pipeline
on:
 push:
 branches: [ main ]
 paths: [ 'src/**' ]
 pull_request:
 branches: [ main ]
 workflow_dispatch:

concurrency:
 group: ci-${{ github.ref }}
 cancel-in-progress: true

jobs:
 test:
 runs-on: ubuntu-latest
 steps:
 - name: Checkout Code
 uses: actions/checkout@v4
 - name: Run Tests
 run: |
 echo "Running unit tests..."
 npm test
 
 deploy:
 needs: test
 if: github.ref == 'refs/heads/main'
 runs-on: ubuntu-latest
 steps:
 - name: Deploy to Cloud
 run: echo "Deploying version ${{ github.sha }} to Prod..."
```

7. STAFF-LEVEL PRO TIPS (PART 1) -- - **Version Pinning**: Hamesha `actions/checkout@v4` jaise version tag use karein. `@main` use karna dangerous hai kyunki bug aa sakta hai.
- **Check Paths**: Har workflow mein `paths` filter lagayein taaki dummy changes (ReadMe) par CI na chale.
- **Timeout**: Har job par `timeout-minutes: 10` lagayein taaki hung process bill na badhaye.

2. CODE: REUSABLE CI WORKFLOW -- ```yaml
# FILE: .github/workflows/reusable-ci.yml
name: Reusable CI
on:
 workflow_call:
 inputs:
 node_version:
 required: true
 type: string
 outputs:
 artifact_id:
 value: ${{ jobs.build.outputs.art_id }}

jobs:
 build:
 runs-on: ubuntu-latest
 outputs:
 art_id: ${{ steps.save.outputs.id }}
 steps:
 - uses: actions/checkout@v4
 - name: Setup Node
 uses: actions/setup-node@v4
 with:
 node-version: ${{ inputs.node_version }}
 - id: save
 run: echo "id=build-${{ github.sha }}" >> $GITHUB_OUTPUT
```
